# v3.4.0 — Histórico Dry-Run

## Objetivo

Escanear Gmail desde `2025-01-01` por trimestres para generar un inventario completo de facturas **sin tocar ningún dato real**. Útil para auditar el pasado antes de activar la automatización completa.

## Garantías absolutas

| Acción prohibida | Estado |
|---|---|
| `appendRow` en FACTURA PROVEEDORES | ✗ nunca |
| Subir PDF a Google Drive | ✗ nunca |
| Modificar etiquetas Gmail | ✗ nunca |
| Modificar MAESTRO_PROVEEDORES | ✗ nunca |
| Escritura en cualquier Sheet | ✗ nunca |

Solo lectura: Gmail (mensajes + adjuntos en RAM) y Sheets (MAESTRO_PROVEEDORES).

## Uso

```powershell
# Desde PowerShell en Windows — requiere token.pickle válido
cd C:\ClaudeProyectos\Bordagran\gmail-facturas-bordagran

# Todos los trimestres (defecto)

# Rango custom
python scripts\generar_historico_dryrun.py --skill-dir . --desde 2025-01-01 --hasta 2025-06-30
```

## Archivos generados

### `runtime/historico_facturas_dryrun.json`
Resultado completo estructurado. Incluye:
- Metadatos (`_meta.version`, `_meta.generado`)
- Array `trimestres[]` con stats y docs de cada trimestre
- `resumen_global` con totales por estado, tipo y proveedor

### `runtime/tipos_documento_candidatos.csv`
Un registro por PDF procesado. Columnas:

| Columna | Descripción |
|---|---|
| `trimestre` | Q1-2025, Q2-2025, … |
| `fecha_email` | Fecha de recepción del email |
| `fecha_pdf` | Fecha extraída del PDF (si disponible) |
| `remitente` | Email del remitente |
| `proveedor` | Nombre normalizado del proveedor |
| `tipo_documento` | factura / abono / no_fiscal / pedido_cliente / … |
| `estado_estimado` | Registrada / Revisar / No fiscal (tipo) / Excluido (Maestro) |
| `num_factura` | Número extraído del PDF |
| `total` | Importe total |
| `iva_pct` | Porcentaje IVA detectado |
| `base` | Base imponible |
| `nombre_pdf` | Nombre del archivo adjunto |
| `msg_id` | ID del mensaje Gmail (para trazabilidad) |
| `notas` | Notas del parser / advertencias |

### `runtime/proveedores_candidatos.csv`
Proveedores **no reconocidos** en `proveedores.json` — un registro por remitente único. Columnas:

| Columna | Descripción |
|---|---|
| `trimestre` | Trimestre de primer avistamiento |
| `fecha_email` | Fecha del primer email |
| `remitente` | Email completo |
| `nombre_detectado` | Nombre derivado del dominio |
| `nombre_pdf` | Nombre del adjunto |
| `msg_id` | ID Gmail |
| `motivo` | Siempre: "Proveedor no reconocido en proveedores.json" |

Los candidatos de este CSV son entradas candidatas para añadir a `references/proveedores.json` o al `MAESTRO_PROVEEDORES`.

## Lógica de clasificación

El script reutiliza las funciones de `procesar_facturas.py` vía import:

```
identificar_proveedor()      → proveedor reconocido o _desconocido
es_email_excluido()          → exclusiones de exclusiones.json
clasificar_tipo_documento()  → factura / abono / no_fiscal / …
cargar_maestro_proveedores() → (solo lectura, para estado_estimado)
buscar_en_maestro()          → lookup normalizado
criterio_maestro()           → instrucción del Maestro
determinar_estado()          → estado fiscal estimado
```

### Orden de prioridad para `estado_estimado`

1. Email excluido → skip (no aparece en CSV)
2. Proveedor desconocido → `Revisar`
3. Tipo documento no fiscal → `No fiscal (tipo)`
4. Excluido en Maestro → `Excluido (Maestro)`
5. Proveedor nuevo (Maestro disponible pero sin entrada) → `Revisar [nuevo en Maestro]`
6. Criterio Maestro activo → aplica instrucción
7. Lógica fiscal v3.2.1 → `Registrada` / `Revisar`

**Nota:** el estado es una **estimación**. El dry-run no registra en MAESTRO_PROVEEDORES a los proveedores nuevos (a diferencia del modo real). Tampoco aplica deduplicación por `clave_unica`.

## Compatibilidad

- v3.3.0 queda **intacta** — `procesar_facturas.py` no se modifica
- `generar_historico_dryrun.py` importa de `procesar_facturas.py` vía `sys.path`
- Si `MAESTRO_PROVEEDORES` no existe → degrada silenciosamente (mismo comportamiento que v3.2.1)
- Si `token.pickle` no existe o ha expirado → se pide re-autenticación por navegador (igual que siempre)
